## 3. Chronological Development Record

### 3.1 Early Need: “Make This Into Something Useful”

The earliest meaningful problem was not “write a cooler prompt.” It was that the user had a large, messy, contradictory pile of chats, ideas, prompt drafts, build attempts, Fable/Hermes thoughts, JARVIS inspiration, dashboard concepts, recovery ideas, and worker-agent plans.

The user wanted the useful truth extracted from that mess, not three different prompt variants or polished noise.

This established the first archive rule:

> Preserve intent over wording.

### 3.2 Discovery: The Pack Was Strong but Dangerous as a Giant Prompt

The early Fable Pack review identified that the source material had a good engine inside it: ground first, find the real wound, merge before rebuild, verify hard, use barbell routing, preserve trust, and create durable rails. But it also had repeated sections, overlapping prompt forms, hype language, behavioural-biopsy material, and too much context bloat.

The correction was:

> Do not run the big pack as-is. Use it as source evidence. Compress it into kernel rules, skills, worker contracts, verification gates, and recovery rails.

That became the first structural transformation.

### 3.3 The Hermes Correction

A major drift point was over-weighting Hermes. Hermes is valuable because of its self-learning and evolving behaviour, but the user corrected the plan sharply: Hermes should not become the center of the architecture.

The final correction:

> The OS comes first. Hermes runs inside it.

Hermes may observe, capture, propose lessons, suggest skill improvements, and operate approved workflows. It may not silently change policies, promote trusted memory, edit hooks or crons, rewrite prompts, decide architecture, or certify final success.

### 3.4 The Memory Correction

Another major correction was that persistent memory was missing or under-weighted in earlier plans. The final design made memory first-class.

But memory was not allowed to become authority.

The final memory principle:

> Memory is evidence, not authority.

Persistent memory, scratch memory, quarantine memory, session summaries, path registry, source summaries, issues, decisions, skill history, automation history, and validation history are all important. But none of them may grant permissions, override Kernel law, or promote themselves into trusted state.

### 3.5 The Intake Failure

A key process failure was discovered: the agent was being asked to process before all source material had been received. This led to premature architecture and fragments.

The correction was the WAIT MODE / PROCESS NOW split:

- WAIT MODE: acknowledge received source parts only.
- PROCESS NOW: only after source intake is complete, begin distillation.

This became part of the source-ingestion protocol and must be preserved for future large updates.

### 3.6 The Final Handoff Pivot

At first, the handoff prompt asked Fable 5 to produce a full Factory Pack. Later, the system was hardened: asking Fable to build the entire OS at once was still too broad.

The final pivot was:

> Fable 5 must build Phase 0 only.

The immediate goal became Source Lock, not Kernel implementation.

## 11. Authority Laundering Rule

Authority laundering through usefulness is the most dangerous failure mode.

A task may discover a problem.  
A worker may recommend a fix.  
A skill may describe a procedure.  
A validator may confirm a finding.  
A recovery rail may propose restoration.  
The UI may display an approval gate.

None of that is permission.

Usefulness is not authority.  
Correctness is not authority.  
Obviousness is not authority.  
Urgency is not authority.  
Past user frustration is not authority.  
Previous approval is not current approval.  
A validated finding is not permission to repair.

A task may mutate state only after passing the full Kernel chain:

Task Card → Router Decision → Permission Decision → Capability Lease → Audit Event → Independent Validation → Operator Approval → Mutation → Post-Validation.

Any component that turns “I found the issue” into “I fixed it” without that chain fails the OS.

- schema validity
- identity
- canonical path safety
- permission scope
- least privilege
- capability/contract match
- command safety
- credential safety
- dependency existence
- conflict check

- environment fit

- TRUSTED_READABLE

## 17. Active Integrity Tests

Static validation proves the configuration looks safe. Dynamic tests prove the system behaves safely.

The OS must run active integrity tests before live capability promotion.

Core tests:

- permission denial test
- path escape test
- secret redaction test
- command filter test
- hallucination detection test
- contract enforcement test
- quarantine behaviour test
- logging integrity test
- approval gate test
- recovery halt test

If any critical test fails, the OS remains in SAFE STATE.

The purpose is to pull the handle, kick the door, try the wrong key, and check the alarm before letting the building wake up.

Task Card → Router Lane Decision → Kernel Permission Decision → Worker Capability Lease → Skill Invocation → Audit Event Before Execution → Skill Output → Independent Validation → Audit Event After Execution → Kernel Promotion or Rejection.

The phrase that survived:

> A Skill is a blade without a handle.

## 24. Phase 8: Recovery Rails

Recovery must not become a backdoor repair bot.

The OS distinguishes:

- Emergency Halt
- Restoration

### 24.1 Emergency Halt

Emergency Halt is a reflex.

It may remove authority:

- stop task execution
- revoke capability leases
- pause workers
- block dispatch
- disable skills
- freeze promotion
- preserve evidence
- emit audit event or failure record
- mark DEGRADED_SAFE_STATE
- notify operator

It may not repair, delete, overwrite, restart, edit, clean, or fix.

### 24.2 Restoration

Restoration is a controlled procedure.

Restoration requires:

- exact failure classification
- preserved evidence bundle
- causal audit trace
- exact target boundary
- minimal mutation plan
- reversibility proof
- independent validation
- human approval for mutation
- recovery capability lease
- post-restoration validation

Emergency Halt can pull power. Restoration must prove it deserves a single insulated tool.

## 41. Glossary

### Agent OS

A control system governing agents, models, memory, tasks, permissions, skills, automation, validation, logs, recovery, and operator visibility.

### SAFE_STATE

Readonly/default-deny boot state where no live mutation, shell execution, destructive action, network action, credential exposure, or autonomous action is allowed without Kernel permission.

### Kernel

The law layer of the OS. It owns SAFE_STATE, permissions, validation gates, capability promotion, approval boundaries, and halts.

### Registry

A source of configuration testimony. It can describe available resources but cannot grant authority.

### Memory

Evidence store. It contains source, project, scratch, agent, audit, and preference records, but cannot authorize action.

### Router

The airlock between request and execution. It selects safe task lanes.

### Task Card

The explicit contract for a unit of work: purpose, target, allowed/forbidden actions, required output, validation, rollback, and evidence.

### Capability Lease

Temporary scoped permission granted by Kernel for one task, one actor, one boundary, one output, and one audit chain.

### Worker

A bounded executor that performs task-card work under a lease.

### Skill

A reusable procedure or recipe that cannot run without a Task Card and Kernel authorization.

### Validator

An independent checker that validates executor output against evidence and policy.

### Audit Ledger

Append-only causal event chain recording what happened, why, under whose authority, and with what result.

### Emergency Halt

Immediate removal of authority to contain risk.

### Restoration

Controlled recovery process after halt, requiring proof, rollback, validation, audit, and approval.

### Command Cockpit

Operator interface showing verified OS state, tasks, approvals, leases, alerts, evidence, and recovery posture.

### Authority Laundering

The unsafe conversion of a useful finding or recommendation into unauthorized action.
